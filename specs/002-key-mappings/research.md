# Research Notes: 按键映射功能选型与决策

## 1. 核心决策与技术路线

### 决策 1: 按键映射的标识符唯一性方案
* **决定**: 内建的空映射使用常量字符串 `"none"`，其余自定义和预设条目统一在 PC 端创建时生成全局唯一的 UUIDv4 标识符。
* **理由**: 手机网页端在选择按键映射后，会使用 LocalStorage 进行持久化缓存，以便下次连入时回显。如果使用顺序索引 (0, 1, 2) 或易变的显示名称，当用户在 PC 端删除、插入或改名时，手机端发送的旧 ID 会导致 PC 端模拟错误的按键动作。UUIDv4 确保了即使配置发生任何变更，新旧 ID 也绝对不会重合。此外，为了维持手机端界面的美观和规整，我们对显示名称强制限制为最大 12 个字符（中英文字符均可，避免被撑开），且在 PC 端保存时强制进行唯一性校验（不可重名）。双层校验机制大大增强了系统的安全和容错能力。
* **备选方案**:
  * *显示名称作为 ID*: 如果用户改名，旧缓存失效；可能重名。
  * *数组索引作为 ID*: 顺序改变后直接导致错乱（已否决）。

### 决策 2: 手机网页端的数据同步与热更新
* **决定**: 基于长连接 WebSocket 的主动广播局部热更新（Local Hot Reload），辅以 `reload` 控制协议进行强制整页重载。
* **理由**: 用户在使用手机输入时，可能在 PC 端新增映射。若采用整页重载 (Reload) 方式更新，手机的 WebSocket 将闪断、聊天记录重绘且中断识别流程。基于 WebSocket 传输最新的配置列表进行局部 DOM 替换可以实现无感知同步。
* **备选方案**:
  * *整页重载 (Reload)*: 能够重置所有状态，但频繁使用体验极差。现保留作为系统重置的兜底机制，不作为常规同步手段。
  * *定时轮询 (Polling)*: 网页端每隔几秒轮询一次配置，会导致局域网无意义的网络开销与延迟，且费电。

### 决策 3: 配置数据的持久化方案
* **决定**: 复用已有的单例 `SettingsManager`，在 `settings.json` 内以独立的 JSON 数组保存所有的按键映射条目。
* **理由**: 该软件是一个桌面微型工具，不建议引入 SQLite 或关系型数据库，配置极其轻量，用 `json` 文件保存完全满足读写性能和规模需要。
* **备选方案**:
  * *使用 SQLite*: 增加了额外依赖，增加了打包复杂度。

## 2. 核心研发原则对齐
* **解耦 (Principle I)**: UI 界面与核心逻辑彻底分离。PC 端的 `KeyMappingsManager` 是纯 Python 类，不依赖任何 PySide6 的 GUI 控件，支持无 GUI 状态下的直接实例化和测试。
* **可打桩性 (Principle II)**: 本地单元测试时，模拟按键组件（`pyautogui`/`pywin32`）会被完全 Mock，使测试能在 Linux/GitHub Actions 无 GUI 的 headless 环境下跑通。
* **防注入安全性 (Principle IV)**: 所有的按键序列在 PC 端保存时，使用 `keyboard.py` 中的 `validate_key_sequence` 进行严格的合法按键格式和长度校验，绝不接受任何非标准按键字符，防止产生系统提权或命令行执行攻击风险。
