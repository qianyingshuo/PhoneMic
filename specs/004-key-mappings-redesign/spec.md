# Feature Specification: 手机端双按键映射通道重构 (key-mappings-redesign)

**Feature Branch**: `004-key-mappings-redesign`

**Created**: 2026-07-01

**Status**: Draft

**Input**: User description: "把发送追加按键映射与单独发送按键解耦。手机端主界面提供两个独立下拉框，分别设置『发送时追加』和『单独发按键』。当输入框非空时，发送文本并追加『发送时追加』的按键映射；当输入框为空时，点击发送（或按键盘回车）仅单独触发『单独发按键』代表的按键。去掉主界面单独的火箭发射按钮，移除冗余的自动追加提示块。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 文本发送与追加解耦 (Priority: P1)

用户在使用 PhoneMic 进行语音打字时，不需要担心因为之前单独模拟过某个按键（例如 `Tab`）而导致接下来的所有文本发送都会自动追加该按键。

**Why this priority**: 核心高频场景。用户经常需要在纯文本打字（不追加任何按键）和文本后自动追加回车/换行（例如在聊天软件中发送消息）之间无缝切换。

**Independent Test**: 用户将“发送时追加”设为“无 (不追加)”，在输入框输入文本并发送，电脑端正常录入文本且光标不发生任何额外跳动。

**Acceptance Scenarios**:

1. **Given** 手机端与 PC 端已正常连接，**When** 用户将“发送时追加”选择为“无”，并在输入框中输入 "你好" 点击发送，**Then** PC 端光标处仅打出 "你好"，不追加任何按键。
2. **Given** 手机端与 PC 端已正常连接，**When** 用户将“发送时追加”选择为“回车 (enter)”，并在输入框中输入 "搜索内容" 点击发送，**Then** PC 端光标处打出 "搜索内容"，并在末尾触发一次回车（如在浏览器搜索框中直接触发搜索）。

---

### User Story 2 - 空输入快速触发单独按键 (Priority: P1)

当用户需要单独控制电脑上的某些按键（如确认、跳格、删除）时，如果输入框为空，直接点击右下角大大的“发送”按钮，或者敲击手机输入法键盘的回车/发送键，就能直接控制电脑。

**Why this priority**: 高效的交互补充。将无文字状态下的空发送语义重定向为按键模拟，极大地提升了用户盲操和单手控制电脑按键的体验。

**Independent Test**: 用户清空输入框，将“单独发按键”设为 `Tab (tab)`，点击绿色发送按钮，电脑光标自动向前跳一格。

**Acceptance Scenarios**:

1. **Given** 输入框为空且“单独发按键”设为 `Tab (tab)`，**When** 用户点击绿色的『发送』按钮或在手机键盘按下发送，**Then** 电脑端只模拟按下一遍 `Tab` 键，网页的聊天历史中追加回显 `[单独发送按键] Tab (tab)`。
2. **Given** 输入框为空且“单独发按键”设为 `无 (none)`，**When** 用户点击绿色的『发送』按钮，**Then** 系统自动拦截该操作，页面不发生任何数据传输，也不会向电脑端发送无意义消息。

---

### User Story 3 - 界面去噪与控制台简化 (Priority: P2)

手机端界面需要去掉所有对用户心智产生干扰的冗余控件，比如单独的火箭发射按钮和“输入时自动追加”灰色提示条，使面板清爽明了。

**Why this priority**: 提升视觉与操作满意度，消除移动端局促屏幕下的臃肿感。

**Independent Test**: 打开手机端主页面，底部快捷控制栏仅呈现两个下拉框，没有任何多余的按钮或修饰文本。

**Acceptance Scenarios**:

1. **Given** 打开手机端主界面，**When** 展开左上角侧边栏（Drawer），**Then** 原本的按键映射选项已成功移出，界面显示相关的迁移说明。
2. **Given** 查看手机端主页，**When** 定位到快捷面板时，**Then** 界面仅能看到并行的“发送时追加”和“单独发按键”两组下拉框，上一版的 `🚀 单独发送` 物理按钮已从 DOM 中完全移除。

---

### Edge Cases

- **全空格输入**：如果用户在输入框中仅输入了空格或换行（如 `   `），点击发送时，系统应自动剔除多余空白并将该输入判定为“空输入”，进而触发“单独发按键”的通道逻辑，而不是将空格文本发送给电脑端。
- **离线与可用性边界**：若局域网连接断开（展示连接断开状态栏时），追加下拉框、单独发送下拉框以及底部输入框、发送按钮统一被禁用（`disabled = true`）。重新连接后自动恢复可用。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 手机端主界面必须提供两个相互独立的选择下拉框：
  - `key-mapping-select`（“发送时追加”）
  - `single-key-mapping-select`（“单独发按键” [NEW]）
- **FR-002**: 手机端发送文本时（无论通过点击底栏发送键还是键盘回车），必须在前台执行非空条件分支判断。若输入非空，使用 `key-mapping-select` 绑定的按键并发送文本；若输入为空，使用 `single-key-mapping-select` 绑定的按键发送空文本。
- **FR-003**: 手机端必须限制无效空发送行为。当输入框为空且 `single-key-mapping-select` 的值为 `none` 时，系统必须拦截发送动作，且不传输任何 WebSocket 消息。
- **FR-004**: 当单独触发按键映射发送时（空文本场景），手机端应在本地聊天列表中追加显示带有特定前缀的回显消息，格式定义为：`[单独发送按键] 显示名称 (按键序列)`。
- **FR-005**: 手机端界面必须完全移除之前版本加入的 `btn-send-mapping` 元素以及对应的文本块。
- **FR-006**: 按钮文字应保持静态的 `发送` / `自动`，不得随当前所选按键动态跳变。
- **FR-007**: 手机端必须为两个下拉选择框提供独立的本地持久化缓存（LocalStorage 键名分别定义为 `selected_key_mapping_id` 和 `selected_single_key_mapping_id`）。当用户修改选择时应自动更新对应的本地缓存，且在页面重载时应自动读取并恢复上一次的选择状态。

### Key Entities

- **发送时追加通道 (Append Key Channel)**: 控制发送字词时附加模拟按键，用于自动上屏后的控制（常驻设置）。
- **单独发送通道 (Single Key Channel)**: 控制无字状态下单独模拟按键，用于远程方向、回车、删除等单按键远程控制（即时触发）。

---

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户因为单独模拟按键而导致正常文字发送被意外追加按键的概率降低到 0%。
- **SC-002**: 用户清空输入框时，点击发送键或手机回车键触发按键模拟的操作响应成功率达到 100%。
- **SC-003**: 去除无用元素后，手机页面下方快捷操作栏的高度不应超过 `100px`，充分保证聊天气泡可见面积。
